## JRA競馬投票用Jsonファイル構造

{
  "version": "1.3",
  "sender": "T58_Box_v20",
  "request_id": "REQ-20260322-COMBINED-001",
  "timestamp": "2026-03-22T10:00:00+09:00",
  "orders": [
    {
      "order_id": "ORD-001",
      "race_id": "20260322060211",
      "post_time": "15:40",
      "ticket_type": "win5",
      "win5_details": {
        "select_n1": [3, 5],
        "select_n2": [1, 8, 10],
        "select_n3": [2],
        "select_n4": [7, 12],
        "select_n5": [1, 15]
      },
      "unit_amount": 100,
      "total_combinations": 24,
      "amount": 2400,
      "expected_ev": 1.8
    },
    {
      "order_id": "ORD-002",
      "race_id": "20260322090212",
      "post_time": "16:25",
      "horse_no": 5,
      "ticket_type": "tan",
      "amount": 1000,
      "expected_ev": 1.4,
      "min_odds": 4.5
    }
  ]
}

### 解説

作成した **JSON仕様 v1.3** の各項目が、**予想側**から**OmniPost**へ何を伝え、どのような判断をさせているのか、その「意味」を詳しく解説します。

この構造により、予想ロジックがどれだけ複雑になっても、投票エンジンは迷わず動けるようになります。

------

### 1. ヘッダーセクション（管理情報）

システム全体を安定させるための「外枠」の情報です。

| **項目名**       | **意味・役割**   | **なぜ必要なのか？**                                         |
| ---------------- | ---------------- | ------------------------------------------------------------ |
| **`version`**    | 仕様のバージョン | 将来、新しい券種や機能が増えた際に、古いプログラムが誤作動しないようにするため。 |
| **`sender`**     | 送信元の名前     | 「どのAIが注文したか」を特定し、プロジェクトごとの成績集計や予算制限を行うため。 |
| **`request_id`** | **重複防止ID**   | 通信エラーなどで同じファイルが2回読み込まれても、**二重投票しないための絶対的なガード**。 |
| **`timestamp`**  | 生成時刻         | システムトラブルで「昨日の予想ファイル」が残っていた場合などに、期限切れとして弾くため。 |

------

### 2. オーダー共通セクション（執行指示）

個別の買い目に関する「いつ・どこで・何を」の情報です。

| **項目名**        | **意味・役割**   | **なぜ必要なのか？**                                         |
| ----------------- | ---------------- | ------------------------------------------------------------ |
| **`order_id`**    | 注文ごとの識別子 | 1つのファイルに複数の買い目がある際、どれが成功してどれが失敗したかを正確にログに残すため |
| **`race_id`**     | 14桁レースコード | 場（中山・阪神など）、日、レース番号を世界で唯一特定するID。 |
| **`post_time`**   | **発送予定時刻** | OmniPostが「あと何分で締め切りか」を画面上の時刻と照らし合わせ、安全に投票するため。 |
| **`ticket_type`** | 券種             | `tan`（単勝）、`win5` など。これによりOmniPostが操作する画面を切り替えます。 |
| **`amount`**      | **投資金額**     | 基準となる金額。OmniPostの「ころがし→単発」判定のベースになります。 |
| **`expected_ev`** | **期待値**       | 残高が少ない時、OmniPostが「期待値が高い順に優先して買う」という賢い判断をするため。 |

------

### 3. WIN5専用セクション (`win5_details`)

WIN5特有の複雑な組み合わせを伝えるための専用設計です。

| **項目名**               | **意味・役割**     | **なぜ必要なのか？**                                         |
| ------------------------ | ------------------ | ------------------------------------------------------------ |
| **`select_n1〜n5`**      | 第1〜5レースの馬番 | 各レースで選んだ馬をリスト形式で指定。IPATの入力画面にそのまま対応します。 |
| **`unit_amount`**        | 1点あたりの金額    | 通常100円。WIN5は点数が多いため、1点単価を明記します。       |
| **`total_combinations`** | 組み合わせ総数     | 全体の点数。OmniPost側でも再計算し、金額に間違いがないかチェック（バリデーション）します。 |

------

### 4. 単勝・通常馬券専用セクション

| **項目名**     | **意味・役割**     | **なぜ必要なのか？**                                         |
| -------------- | ------------------ | ------------------------------------------------------------ |
| **`horse_no`** | 馬番               | 狙い撃ちする馬の番号。                                       |
| **`min_odds`** | **最低保証オッズ** | 投票直前にこれよりオッズが下がっていたら「期待値がない」と判断して購入を中止する防衛策。 |

------

### このJSONが実現すること

このJSONは、いわば**「AI軍師から現場の実行部隊への指令書」**です。

1. **予測側** は、残高や締切時間を気にせず、このJSONを生成するだけで仕事完了。
2. **OmniPost** は、このJSONを読み取り、「今は目標金額に達しているから、指示にある `amount` を減らして単発で買おう」といった**現場の判断**を下します。
